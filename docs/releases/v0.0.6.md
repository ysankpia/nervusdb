# NervusDB v0.0.6

This release is a performance-evidence release for the embedded graph core. It
does not add query features or change public API. It makes benchmark attribution
honest, removes low-risk storage hot-path waste, and records the remaining
storage-layout work for `v0.0.7`.

## What Changed

- Added a cross-database embedded graph benchmark comparing:
  - NervusDB,
  - SQLite with a simple graph schema,
  - SQLite with materialized graph indexes.
- Split benchmark timing so load and reopen costs are attributable:
  - `load_total_ms`,
  - `reopen_open_ms`,
  - `reopen_count_verify_ms`.
- Added `NERVUSDB_PROFILE_STORAGE=1` internal storage profiling.
- Removed property-index cleanup paths that scanned all `idx_node_props` for a
  single node/property cleanup.
- Streamed outgoing and incoming adjacency prefix scans instead of first
  materializing all matching keys into `Vec<Vec<u8>>`.
- Removed normal traversal liveness point reads that were redundant with the
  0.0.3 graph-integrity invariant.
- Removed repeated `created_nodes` linear lookup from bulk property-index writes.
- Skipped old property-index cleanup for nodes created in the same transaction.

## Remaining Work

The easy hot-path bugs are no longer the main problem. The remaining large costs
are durable `batch.commit`, raw database reopen, file count, and LSM storage
footprint. These point to a storage-layout project, not another small patch.

That work is deliberately deferred to `v0.0.7` and must be covered by a storage
layout ADR before implementation.

## Install

```toml
[dependencies]
nervusdb = "0.0.6"
```

Only `nervusdb` is published. The workspace-local crates `nervusdb-api`,
`nervusdb-storage`, and `nervusdb-query` remain `publish = false` wrappers and
are not separate public packages.

## Explicit Non-Goals

This release does not include:

- keyspace merge,
- storage format rewrite,
- unsafe buffered write mode,
- public bulk-loader or durability configuration,
- range indexes,
- public `Db::create_index` or `GraphSnapshot::lookup_index`,
- edge IDs or parallel edges,
- vector/HNSW search,
- multi-writer concurrency,
- full openCypher,
- Python/Node/C bindings,
- long-term storage-format compatibility promises before 0.1.
