# NervusDB v0.0.2

This release keeps the 0.1-core product surface from `v0.0.1` and fixes the
first real post-release bottleneck: bulk graph writes.

It does not add new public APIs, change the storage format contract, disable
durable writes, or revive advanced Cypher features.

## What Changed

- Node id allocation now happens inside `WriteTxn` staging.
- `next_node_id` is persisted in the same atomic Fjall commit batch as the node
  and edge data.
- Edge staging now uses `Vec<EdgeKey>` and commit-time `sort_unstable()` /
  `dedup()` instead of per-edge `BTreeSet` insertion.
- The benchmark driver now reports stage timings for open, schema lookup, node
  creation, edge creation, commit, reopen verification, reads, and write smoke.
- `scripts/core_bench.sh` now names custom benchmark artifacts correctly instead
  of labeling explicit `--nodes/--degree/--iters` runs as `small`.

## Why It Matters

The `v0.0.1` 100k node / 500k edge benchmark took 438.130s to insert. The real
bug was not Fjall and not `SyncAll` durability by itself. The old path called
`next_counter(META_NEXT_NODE_ID)` from every `create_node`, and each counter
increment committed with `PersistMode::SyncAll`.

That meant 100k nodes caused 100k durable metadata commits before the real
transaction commit. `v0.0.2` fixes the transaction boundary: normal commits
still use `SyncAll`, but node id metadata is committed once with the transaction.

## Install

```toml
[dependencies]
nervusdb = "0.0.2"
```

Only `nervusdb` is published. The workspace-local crates `nervusdb-api`,
`nervusdb-storage`, and `nervusdb-query` remain `publish = false` wrappers and
are not separate public packages.

bash scripts/core_examples.sh

cargo publish -p nervusdb --dry-run --registry crates-io
```

GitHub Actions for `main` passed for the write-path commit and validation
recording commit.

```

Comparison:

| Metric | v0.0.1 | v0.0.2 | Result |
|---|---:|---:|---:|
| Insert time | 438.130s | 0.415s | 1055x faster |
| Insert edges/sec | 1,141 | 1,204,516 | 1056x higher |
| Write txn p99 | 12.6305ms | 4.0340ms | 68% lower |

Repeated 100k/500k runs stayed far above the 2x target:

- `artifacts/core-bench/core-bench-custom-100000n-5d-20260621-190709.json`
  inserted at 881,240 edges/sec.
- `artifacts/core-bench/core-bench-custom-100000n-5d-20260621-190713.json`
  inserted at 1,043,851 edges/sec.

Read throughput varied across runs. The release does not claim a new read
performance baseline; it claims the write-path bug is fixed without changing the
read-path layout.

## Explicit Non-Goals

This release still does not include:

- property indexes,
- old `.ndb + .wal` compatibility,
- full openCypher,
- parallel edges or edge ids,
- public bulk loader / durability options,
- unsafe buffered write mode,
- Python/Node/C bindings,
- vector/HNSW search.
