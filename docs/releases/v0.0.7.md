# NervusDB v0.0.7

This release is a storage-format cleanup release for the embedded graph core. It
does not add query features or change the public Rust API. It deliberately
breaks on-disk compatibility to move NervusDB to storage epoch 3.

The point of this release is narrow:

- make clean reopen fast,
- reduce storage footprint,
- keep fsck and crash recovery working on the new layout,
- document the remaining performance limits honestly.

## Breaking Storage Change

`STORAGE_FORMAT_EPOCH` is now `3`.

NervusDB `v0.0.7` does not open `v0.0.6` database directories. Old directories
return `StorageFormatMismatch`. There is no migration tool in this release.
Users should rebuild, recreate, or export/reimport data.

## What Changed

- Added ADR 0009 for epoch 3 storage keyspace consolidation.
- Replaced the old many-keyspace physical layout with four Fjall keyspaces:
  - `meta`,
  - `graph_data`,
  - `adj_out`,
  - `adj_in`.
- Added a tagged internal `graph_data` layout for non-adjacency graph records:
  nodes, external-id mapping, label/relationship names, node labels, label scan
  index, node properties, edge properties, and node property equality index.
- Kept adjacency in dedicated `adj_out` and `adj_in` keyspaces to preserve a
  clear hot traversal path.
- Changed `GraphEngine::open` to validate `meta/format_epoch` before opening
  graph keyspaces, so rejected old directories are not polluted with new epoch 3
  keyspaces.
- Changed explicit `Db::close()` to perform a clean shutdown flush after
  journal persistence. This avoids heavy journal replay on clean reopen.
- Updated fsck-lite to scan the epoch 3 layout.
- Added storage tests for:
  - epoch 2 rejection,
  - epoch 3 physical keyspace count,
  - existing graph invariants on the new layout.

## Interpretation

This release succeeds as a clean-reopen and footprint release. Clean reopen
moves from seconds to milliseconds, and disk footprint drops by more than half.

It is not a universal performance release. Traversal throughput regressed versus
the v0.0.6 benchmark line. That regression is documented, not hidden. Do not
treat v0.0.7 as proof that keyspace consolidation solves traversal speed.

Future traversal work should be benchmark-first and should not start by blindly
changing keyspace counts again.

## Install

```toml
[dependencies]
nervusdb = "0.0.7"
```

Only `nervusdb` is published. The workspace-local crates `nervusdb-api`,
`nervusdb-storage`, and `nervusdb-query` remain `publish = false` wrappers and
are not separate public packages.

## Explicit Non-Goals

This release does not include:

- old database migration,
- single-file storage,
- unsafe buffered write mode,
- public bulk-loader or durability configuration,
- traversal repair beyond the documented layout work,
- range indexes,
- public `Db::create_index` or `GraphSnapshot::lookup_index`,
- edge IDs or parallel edges,
- vector/HNSW search,
- multi-writer concurrency,
- full openCypher,
- Python/Node/C bindings,
- long-term storage-format compatibility promises before 0.1.
