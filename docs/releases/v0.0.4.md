# NervusDB v0.0.4

This release adds the first practical query-anchoring index for the 0.1-core
line: an internally maintained node property equality index.

It does not add public index-management APIs, range indexes, edge property
indexes, composite indexes, unique constraints, edge IDs, parallel edges,
vectors, multi-writer concurrency, unsafe write modes, or broader Cypher.

## What Changed

- Added internal `idx_node_props` storage keyspace for label-qualified exact
  node property equality lookup.
- Added storage-neutral `GraphSnapshot::nodes_with_label_and_property`.
  The default implementation scans `nodes_with_label`; the Fjall-backed
  snapshot uses `idx_node_props`.
- Maintained node property index entries atomically in the same commit batch as
  graph writes.
- Covered index maintenance for node property set/remove, label add/remove,
  node tombstone cleanup, same-transaction final state, and reopen consistency.
- Made Mini-Cypher anchor these query shapes through the indexed snapshot path:

```cypher
MATCH (n:Label) WHERE n.key = 'value' RETURN n
MATCH (n:Label {key: 'value'}) RETURN n
```

- Kept the regular `Filter` plan above the indexed scan as a correctness guard
  for multiple predicates and unsupported downshift cases.
- Extended the core benchmark to report scan-baseline versus indexed property
  lookup timing.

## Why It Matters

`v0.0.3` made graph integrity enforceable before index work. That ordering
matters: an index on top of stale labels, dangling edges, or half-deleted nodes
would make wrong answers faster.

`v0.0.4` uses the now-cleaner write path to maintain a small, honest index. It
does one thing: accelerate label-qualified exact node property lookup. The index
is internal and automatic; users do not manage schema or call `create_index`.

## Benchmark Evidence

Validated on the 100k-node / 500k-edge benchmark:

```text
artifact: artifacts/core-bench/core-bench-custom-100000n-5d-20260622-050241.json
scan baseline: 68,519.803 ms
indexed lookup: 1.435 ms
speedup: 47,757.312x
insert throughput with index maintenance: 222,707.841 edges/sec
hot neighbor traversal: 1,761,907 edges/sec
cold neighbor traversal: 837,486 edges/sec
```

The benchmark compares a manual label scan plus property filter against
`nodes_with_label_and_property`. It does not fake the baseline through the
indexed path.

## Install

```toml
[dependencies]
nervusdb = "0.0.4"
```

Only `nervusdb` is published. The workspace-local crates `nervusdb-api`,
`nervusdb-storage`, and `nervusdb-query` remain `publish = false` wrappers and
are not separate public packages.

## Compatibility Notes

This is a pre-0.1 line and intentionally allows internal API and storage layout
changes. `v0.0.4` adds a new internal logical keyspace, `idx_node_props`, to the
Fjall-backed database directory.

There is still no `.ndb + .wal` public storage format, no old-format migration,
no public index-management API, no range index semantics, no edge property
index, and no parallel-edge identity.

## Explicit Non-Goals

This release still does not include:

- public `Db::create_index` or `GraphSnapshot::lookup_index`,
- range indexes,
- edge property indexes,
- composite indexes,
- unique constraints,
- parameter-based index planning,
- full openCypher,
- parallel edges or edge IDs,
- public bulk loader / durability options,
- unsafe buffered write mode,
- Python/Node/C bindings,
- vector/HNSW search.
