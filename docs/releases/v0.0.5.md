# NervusDB v0.0.5

This release is the stability-freeze line for the current 0.x graph core. It
does not expand the database feature set. It adds a conservative offline
fsck-lite tool and a realistic Agent Memory smoke so NervusDB can be used as a
dependency in downstream projects without continuing active database work by
default.

## What Changed

- Added `nervusdb v2 fsck --db <path>`.
- Added `nervusdb v2 fsck --db <path> --repair`.
- Added `nervusdb v2 fsck --db <path> --json`.
- Added feature-gated `nervusdb::admin` behind `unstable-admin`.
- Added fsck reports with stable top-level JSON fields: `ok`, `repaired`,
  `checked`, `issues`, and `repairs`.
- Added repair for rebuildable derived indexes only:
  - `label_nodes`
  - `idx_node_props`
- Added Agent Memory smoke covering:
  - `Character`, `Event`, and `Fact` nodes,
  - `KNOWS`, `APPEARS_IN`, `CAUSES`, and `MENTIONS` relationships,
  - property-indexed lookup,
  - neighbor traversal,
  - property update,
  - `DETACH DELETE`,
  - drop/reopen consistency,
  - feature-gated fsck on the resulting database.

## Fsck Semantics

Fsck checks the graph storage shape and derived indexes:

- live node labels must match `label_nodes`,
- scalar node properties plus labels must match `idx_node_props`,
- stale derived index entries are reported,
- adjacency symmetry is checked,
- edge properties must point at visible edges,
- node properties must point at live nodes.

Repair is intentionally conservative. It rebuilds `label_nodes` and
`idx_node_props` from canonical graph keyspaces using a durable Fjall batch. It
does not delete canonical user graph data.

Fsck reports but does not automatically repair:

- adjacency mismatch,
- orphan edge properties,
- orphan node properties,
- malformed canonical records.

Run repair offline. Do not run `--repair` while another process writes the same
database directory.

## Why It Matters

`v0.0.4` made exact node property lookup fast. `v0.0.5` makes that indexed path
operationally checkable without adding a public index-management API.

This is also the stop-line release. After `v0.0.5`, new database work should be
driven by real downstream blockers, not by speculative engine expansion.

## Install

```toml
[dependencies]
nervusdb = "0.0.5"
```

Only `nervusdb` is published. The workspace-local crates `nervusdb-api`,
`nervusdb-storage`, and `nervusdb-query` remain `publish = false` wrappers and
are not separate public packages.

## CLI Examples

```bash
nervusdb v2 fsck --db ./graph
nervusdb v2 fsck --db ./graph --json
nervusdb v2 fsck --db ./graph --repair --json
```

Exit codes:

- `0`: database passed fsck.
- `1`: fsck found unresolved issues, or repair finished but issues remain.
- `2`: parameter, open, storage, or I/O error.

## Explicit Non-Goals

This release does not include:

- range indexes,
- public `Db::create_index` or `GraphSnapshot::lookup_index`,
- edge property indexes,
- composite indexes,
- unique constraints,
- edge IDs or parallel edges,
- vector/HNSW search,
- multi-writer concurrency,
- unsafe buffered write mode,
- public bulk-loader or durability configuration,
- full openCypher,
- Python/Node/C bindings,
- long-term storage-format compatibility promises before 0.1.
