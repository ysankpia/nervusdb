# Plan: Core 0.1 Slimdown

## Status

In progress

## Goal

Stop the repository from treating platform breadth as progress. NervusDB 0.1 is
the embedded Rust graph database line: local files, WAL recovery, graph
persistence, one-hop/two-hop traversal, Mini-Cypher, and a CLI for local smoke
usage.

## 30-Day Freeze

Until this plan is closed or superseded, do not add:

- new full-Cypher features
- new Python, Node.js, or C API surface
- new vector/HNSW default-path behavior
- new optimizer capability outside Mini-Cypher
- new release, nightly, parity, soak, chaos, fuzz, or perf gate pressure

Build fixes and security fixes in frozen areas are allowed. New capability work
in those areas needs an ADR first.

## Layers

Core:

- `nervusdb`
- `nervusdb-api`
- `nervusdb-storage`
- Mini-Cypher path in `nervusdb-query`
- `nervusdb-cli` open/query/write/import-smoke subset

Experimental:

- `nervusdb-pyo3`
- `nervusdb-node`
- `nervusdb-capi`
- `examples-test/`
- HNSW/vector search
- optimizer work outside Mini-Cypher
- backup/vacuum non-core API
- TCK harness
- fuzz targets
- perf/chaos/soak scripts

Graveyard / frozen:

- procedures
- subqueries
- pattern comprehension
- full openCypher semantics
- broad temporal/duration semantics
- complex aggregation
- cross-binding parity gate
- release/stability windows

## Not In Scope

- Physical deletion of advanced code in the first pass.
- Breaking public API removal before the API surface plan classifies it.
- Expanding 0.1 to include platform-era goals.
- Running full historical tests for docs-only or harness-only changes.

## Implementation Order

1. Treat `docs/product/scope-0.1.md`, `docs/product/non-goals.md`, and
   `docs/architecture/workspace-layers.md` as the current constitution.
2. Keep default CI on the core local path.
3. Classify current code/docs as core, experimental, or frozen.
4. Mark the 0.1 Rust API surface without removing public functions.
5. Keep Mini-Cypher reference coverage and the core acceptance suite aligned.
6. Maintain smoke, crash recovery, and benchmark script entry points.
7. Split quick validation from full historical fan-out so ordinary development
   does not spend an hour proving unrelated frozen surfaces.
